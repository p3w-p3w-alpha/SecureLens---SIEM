Part 0 — How to Use Claude Code Effectively for This Project

The Core Workflow

You're going to be working feature-by-feature, and each feature follows this loop:

  1. Prompt Claude Code with a clear, scoped task (one feature at a time)
  2. Review the generated code — read it, understand it, don't just accept blindly
  3. Run it and test visually in the browser
  4. Iterate — fix bugs, refine, then move to the next feature
  5. Commit after each working feature

Project Setup for Claude Code

Before you write a single line of code, create a CLAUDE.md file in your project root. This is the file Claude Code reads automatically to understand your project. It's the single most important thing for effective vibe coding.

Your CLAUDE.md file should contain:

How to Prompt Claude Code — Patterns That Work

Pattern 1: Feature-Scoped Prompts

Always give Claude Code one complete feature per prompt. Not "build the backend" — that's too broad. Not "create a User entity" — that's too narrow and you'll lose context between prompts.

Good prompt example
"Implement JWT authentication for SecureLens. Create: (1) User entity with id, username, email, password_hash, role, created_at fields mapped to a 'users' PostgreSQL table, (2) a /api/v1/auth/register endpoint accepting username, email, password and returning a JWT, (3) a /api/v1/auth/login endpoint accepting email and password and returning a JWT, (4) a JwtAuthFilter that validates the token on all /api/v1/** routes except /api/v1/auth/**, (5) a SecurityConfig that wires it all together. Use bcrypt for password hashing. The JWT should contain userId, email, and role. Token expiry: 24 hours."
Bad prompt example
"Add user authentication"

Pattern 2: Referencing Your Architecture

You have the SecureLens_Architecture_Reference.html from our previous conversation. When prompting Claude Code for a feature, reference the specific section. You can even paste relevant excerpts into the prompt.

Reference-based prompt example
"Implement detection rule R-003 (Privilege Escalation). Here's the spec: Trigger when same user_id generates 3+ 'permission_denied' events followed by a 'permission_granted' event within 5 minutes. Severity: HIGH. The rule should run as a @Scheduled job every 60 seconds, query the logs table, and create an alert in the alerts table with mitre_tactic='TA0004', mitre_technique='T1548'."

Pattern 3: Fix-and-Iterate Prompts

When something breaks (it will), give Claude Code the exact error:

Debug prompt example
"The /api/v1/alerts endpoint returns 403 even though I'm passing a valid JWT. Here's the error from the browser console: [paste error]. Here's what I see in the Spring Boot logs: [paste relevant logs]. Fix the JWT filter — I think the token extraction from the Authorization header might be wrong."

Pattern 4: Frontend-with-Backend Prompts

Since you're going feature-by-feature with visual testing, always ask for the frontend as part of the feature:

Full-feature prompt example
"Now create the React frontend for the login/register feature. Create: (1) a LoginPage with email and password fields that calls POST /api/v1/auth/login, stores the JWT in localStorage, and redirects to /dashboard, (2) a RegisterPage with username, email, password fields that calls POST /api/v1/auth/register, (3) an AuthContext that provides the current user and token to all components, (4) a ProtectedRoute wrapper that redirects to /login if no token exists. Use generic Tailwind styling — just make it functional and clean, no fancy design. I need to test this visually."

Git Discipline

Commit after every working feature. This is your safety net:

git add .
git commit -m "feat: implement JWT auth with login/register pages"
git push

If Claude Code breaks something later, you can always roll back. Name your commits clearly so you can find specific features later.

Running Both Servers During Development

You'll always have two terminals open:

# Terminal 1 — Backend
cd backend
./mvnw spring-boot:run

# Terminal 2 — Frontend
cd frontend
npm start

The React dev server typically runs on localhost:3000 and proxies API calls to Spring Boot on localhost:8080. You'll configure this proxy in Phase 1.

Never let Claude Code generate code for multiple unrelated features in one prompt. One feature = one prompt = one commit. This keeps things manageable and debuggable.

Phase 4 — Log Simulator

4
Log Simulator
Goal: A built-in simulator that generates realistic security logs — both benign and malicious — so you can test detection rules without real data.
Task 4.1 — Simulator Service (Backend)

A service that generates logs matching each of the 8 detection rule patterns, plus normal baseline traffic. The simulator should be configurable: number of logs, time window, ratio of malicious to benign. It should use realistic IP addresses, usernames, and event types. The simulator POSTs logs through the same ingestion API to keep things consistent.

Log scenarios to generate:

  • Normal traffic: Scattered logins, file accesses, network connections with varied IPs and users
  • Brute force (R-001): 5+ login_failed from same source_ip in 5 minutes
  • Impossible travel (R-002): Same user_id login_success from IPs in different countries within 30 minutes (store country in metadata)
  • Privilege escalation (R-003): Same user_id with 3+ permission_denied then permission_granted in 5 minutes
  • Data exfiltration (R-004): Same source_ip with 10+ data_transfer events totaling >100MB in 10 minutes (store bytes in metadata)
  • Port scan (R-005): Same source_ip hitting 20+ distinct destination ports in 2 minutes (store port in metadata)
  • Lateral movement (R-006): Same user_id connecting to 5+ distinct internal IPs in 10 minutes
  • Malware beacon (R-007): Same source_ip making network_connection to same destination_ip at regular intervals (±2s jitter) 10+ times in 30 minutes
  • Off-hours access (R-008): login_success or file_access events between 00:00–05:00 local time for users not flagged as shift workers
Task 4.2 — Simulator API & Frontend Controls

POST /api/v1/simulator/run — accepts config: scenarioTypes (list), logCount, timeWindowMinutes. Returns count of generated logs. Frontend: a SimulatorPage with checkboxes for each scenario, a "Run Simulation" button, and a results summary showing how many logs were generated per scenario.

Claude Code Prompt
"Build the log simulator for SecureLens. This generates realistic security logs for testing detection rules. Backend: (1) LogSimulatorService with a method runSimulation(config) that generates logs and posts them through the existing LogService.ingest(). The config includes: scenarioTypes (list of strings), logCount (int), timeWindowMinutes (int). (2) Generate logs for these scenarios: NORMAL — scattered logins, file accesses, connections from varied IPs/users; BRUTE_FORCE — 5+ login_failed from same sourceIp in 5min; IMPOSSIBLE_TRAVEL — same userIdField login_success from IPs in different geolocations within 30min (store {country: 'US'} and {country: 'RU'} in metadata JSON); PRIVILEGE_ESCALATION — same userIdField with 3+ permission_denied then permission_granted in 5min; DATA_EXFILTRATION — same sourceIp with 10+ data_transfer events totaling >100MB in 10min (store {bytes: N} in metadata); PORT_SCAN — same sourceIp hitting 20+ distinct ports in 2min (store {port: N} in metadata); LATERAL_MOVEMENT — same userIdField connecting to 5+ distinct internal IPs in 10min; MALWARE_BEACON — same sourceIp making network_connection to same destinationIp at regular ~60s intervals 10+ times in 30min; OFF_HOURS — login_success or file_access between 00:00-05:00. (4) Results panel showing how many logs were generated per scenario type. (5) A 'Generate All Scenarios' quick button that selects everything. Add nav link. Generic Tailwind."

Phase 5 — SIEM Detection Engine (8 Rules)

5
SIEM Detection Engine (8 MITRE ATT&CK Rules)
Goal: Scheduled detection jobs that scan logs and create alerts. This is the core of SecureLens — take extra care here.
This is the most important phase. Each rule must be implemented correctly with the exact trigger conditions from the architecture spec. Test each rule individually with the simulator.
Task 5.1 — Alert Entity & Schema

Alert entity: id (Long), rule_id (String, e.g., "R-001"), rule_name, severity, mitre_tactic, mitre_technique, source_ip, user_id_field, description (TEXT), evidence_log_ids (JSON array of log IDs that triggered this alert), status (NEW/INVESTIGATING/RESOLVED/FALSE_POSITIVE), created_at, updated_at, resolved_by (nullable, references app user), ai_triage_result (TEXT, nullable — filled later by Mistral).

Task 5.2 — Detection Rule Framework

Create a DetectionRule interface with method evaluate() that queries recent logs and returns a list of alerts. Create a DetectionEngine service that: runs all rules on a schedule (@Scheduled, every 60 seconds), deduplicates alerts (don't create duplicate alert for same source_ip+rule_id within same time window), saves new alerts to DB.

Task 5.3 — Implement All 8 Rules

Implement each rule as a separate class implementing DetectionRule. Here are the exact specs:

Phase 7 — Threat Intelligence Hub

7
Threat Intelligence Hub
Goal: Query external threat intel APIs and display enriched data. Five integrations: VirusTotal, AbuseIPDB, Shodan, NVD, AlienVault OTX.
Task 7.1 — Threat Intel Cache Entity

Cache responses to avoid hitting API rate limits. Entity: id, query_type (IP/DOMAIN/HASH/CVE), query_value, provider (VIRUSTOTAL/ABUSEIPDB/SHODAN/NVD/OTX), response_data (TEXT — full JSON response), risk_score (Integer, normalized 0-100), cached_at, expires_at. Cache TTL: 1 hour.

Task 7.2 — Provider Service Classes

One service class per provider, each implementing a ThreatIntelProvider interface:

  • VirusTotalService: IP reputation, file hash lookup. API: GET https://www.virustotal.com/api/v3/ip_addresses/{ip}

Phase 9 — AI Investigation Assistant (Chatbot)

9
AI Investigation Assistant (Chatbot)
Goal: A persistent chat panel where analysts ask questions in natural language and get answers grounded in real database data. This is the architectural pattern behind Microsoft Copilot for Security, Elastic AI Assistant, and Splunk AI — built from first principles.
This chatbot is architecturally significant: it transforms SecureLens from a dashboard that presents data into an interface that understands intent. The same data becomes accessible in two modes — structured (pages/tables) and conversational (natural language). It reuses the existing MistralService from Phase 8.
Task 9.1 — The Two-Call Mistral Pattern (Backend Core)

The chatbot uses a two-call pattern to ensure Mistral never hallucinates data:

  1. Call 1 (Tool Selection): Analyst's question + tool definitions → Mistral decides which tool to call and with what parameters
  2. Spring Boot executes the query: Runs the actual database query — Mistral never touches the DB directly
  3. Call 2 (Answer Generation): Real data from the query → Mistral formulates a natural language answer

Create ChatService.java that manages this loop, maintains conversation history per request, and handles multi-tool chaining (some questions need multiple tools called in sequence).

Task 9.2 — The 6 Tool Definitions

Spring Boot describes these tools to Mistral at the start of every conversation. Mistral reads the descriptions and decides which one to call. Spring Boot executes only the chosen tool — never arbitrary SQL.

Tool NameWhat It QueriesKey Parameters
query_eventslogs table — raw log linesevent_type, source_ip, from_time, to_time, limit (max 100)
query_alertsalerts table — detected incidentsstatus, severity, mitre_tactic, source_ip, from_time, limit (max 50)
get_alert_detailSingle alert — full detail including AI triage and threat intelalert_id
get_top_attackersRanked list of most active source IPsmetric (events | alerts), from_time, limit (max 20)
get_event_statsAggregated counts grouped by a fieldgroup_by (source_ip | event_type | severity | hour), from_time, to_time
investigate_iocThreat intel for an IP/domain/hash — uses threat_intel_cacheioc_value, ioc_type (IP | DOMAIN | HASH)

Each tool maps to an existing repository query. No new tables, no new dependencies.

Task 9.3 — Chat Controller & API Endpoint

POST /api/v1/chat — Body: { message: string, history: Message[] } — Returns: { answer: string, toolCalled: string, dataSnapshot: object }

The history array contains previous messages of the current session (maintained in React state on the frontend). This gives Mistral conversation context for follow-up questions. The dataSnapshot contains the raw query results so the frontend can optionally render them as a table alongside the natural language answer.

The chatbot should never be blocked by API latency.

Claude Code Efficiency Tips

Update CLAUDE.md after each phase. Change the "Current Phase" section so Claude Code always knows where you are in the project.
If Claude Code generates something wrong, don't start over. Say: "The X you generated has this problem: [specific issue]. Fix only that — don't touch anything else."
When you're stuck, paste the exact error message. Claude Code works best with specific errors, not "it doesn't work."
After finishing each phase, run git diff --stat to see what changed, then commit with a meaningful message. This is your undo button.
If a prompt is generating too much at once and the result is buggy, split it into two prompts. It's better to do two clean prompts than one messy one.
Keep the backend running while working on frontend features. Test in the browser after every change. Don't wait until the end of a phase to test.
Don't skip the checkpoints. Every checkpoint is a moment where you verify the feature works. If you skip verification and move to the next phase, bugs compound and become much harder to fix later.
