Section 1

What SecureLens is

SecureLens is a unified Security Operations Platform built for SOC analysts. It combines three capabilities that in the real world exist as separate expensive enterprise tools: a SIEM engine for real-time log correlation and alerting, a threat intelligence hub for IOC investigation across five public APIs, and an AI-assisted triage layer powered by Mistral that explains alerts and recommends analyst actions in plain language.

It is deployed as a live web application. Anyone with the URL can log in and use it as a real SOC dashboard.

The two modules feed each other through automated enrichment: when the SIEM correlation engine fires an alert, it automatically triggers a threat intelligence investigation on the source IP. By the time an analyst clicks the alert, the AbuseIPDB score, Shodan port data, and VirusTotal malware score are already populated on the alert card. This integration is what makes SecureLens a platform rather than a collection of tools.

Section 8

Mistral AI integration — exact prompt design

When an alert fires, Spring Boot makes one async call to Mistral's mistral-small-latest model. The call does not block the alert from being stored and pushed via WebSocket — the AI fields populate on the alert card within 3–8 seconds after the alert appears. The response is parsed as JSON and stored in ai_summary and ai_recommendation columns.

Mistral is the deliberate choice over OpenAI: Mistral is a French AI company, Sekoia.io and CyberNext both operate in the French-speaking security ecosystem, and OCP Group has a formal partnership with Mistral AI. This is a signal, not just a technical choice.

Exact prompt template

You are a concise SOC analyst assistant. Given the following security alert,
respond ONLY with a valid JSON object containing exactly these fields:
{
  "summary": "2-3 sentence plain-language explanation of what happened",
  "intent": "likely attacker objective based on the MITRE technique",
  "recommendations": ["action 1", "action 2", "action 3"],
  "verdict": "TRUE_POSITIVE or LIKELY_FALSE_POSITIVE",
  "verdict_reason": "one sentence explaining the verdict"
}

Alert context:
- Rule: {rule_name} ({mitre_id} — {mitre_tactic})
- Source IP: {src_ip}
- Events: {event_count} matching events in {time_window}s
- Time: {alert_time} ({is_business_hours})
- AbuseIPDB score: {abuse_score}/100
- Shodan: {shodan_summary}
- VirusTotal: {vt_summary}

Example output for a brute force alert

{
  "summary": "Between 14:32 and 14:35, IP 185.220.101.3 attempted 47 failed SSH logins
              against your server from a known Tor exit node. This pattern is consistent
              with an automated credential stuffing attack targeting common usernames.",
  "intent": "Gain unauthorized access to the system via valid credentials (T1110 — Brute Force)",
  "recommendations": [
    "Block IP 185.220.101.3 at the firewall immediately",
    "Review all successful SSH authentications from this subnet in the last 24 hours",
    "Enable SSH rate limiting and consider moving SSH to a non-standard port"
  ],
  "verdict": "TRUE_POSITIVE",
  "verdict_reason": "AbuseIPDB score of 98/100 and Tor exit node classification confirm
                     malicious origin with high confidence."
}
If JSON parsing fails, a fallback message is stored: "AI analysis unavailable — please review alert context manually." The alert is never blocked by an AI service failure.
Section 9

Log simulator — four source types

The simulator is a Spring Boot @Scheduled(fixedRate = 200) task generating one event every 200ms in normal mode, and 50 events per second during attack simulation mode. Each source has its own generator class with realistic statistical distributions.

SSH auth generator

Produces events with realistic usernames (admin, root, ubuntu, oracle, pi), source IPs from known attack ranges, and a 95/5 split between failed and successful logins during normal mode. During attack simulation it produces 40+ failures from one IP within 60 seconds followed by one success — triggering rules T1110 and T1078 in sequence.

Firewall generator

Produces ALLOW and DENY events across a realistic port distribution. Normal mode is 70% ALLOW traffic. Attack simulation produces rapid DENY events across 600+ distinct destination ports from one IP within 30 seconds — triggering T1046 (port sweep). After the sweep, DENY count drops sharply — triggering T1562 (firewall evasion detection).

Apache web server generator

Produces realistic HTTP log lines including method, path, status code, response size, and user agent string. Normal mode is 90% 200 responses, 8% 404, 2% 5xx. Attack simulation produces 600+ requests per minute from one IP — triggering T1499 (HTTP flood). Paths include realistic web application endpoints.

Windows event generator

Produces Windows Security Event IDs: 4624 (successful login), 4625 (failed login), 4648 (explicit credential use), and 4720 (user account created). Normal mode produces mostly 4624 events. Attack simulation generates a sequence of 4625 events followed by 4648 — triggering T1548 — and optionally 4720 — triggering T1136.

Pre-seeded data — DataSeeder component

On first boot in demo mode, a DataSeeder Spring component inserts 7 days of realistic synthetic events. This includes a brute force campaign from IP 185.220.101.3 (AbuseIPDB score 98, Tor exit node), a port sweep from 192.168.1.100, after-hours authentication events, and a day of normal baseline traffic. When a recruiter opens the deployed URL they see a populated SOC dashboard immediately.

AI investigation assistant (chatbot)

Section 13

AI investigation assistant — chatbot layer

The AI investigation assistant is an always-available chat panel inside the SecureLens dashboard. Analysts type questions in plain language. The chatbot reads the question, decides what data it needs, fetches it directly from the PostgreSQL database, and responds with a natural language answer grounded in real data.

This is what Microsoft Copilot for Security, Elastic AI Assistant, and Splunk AI do at their core. You are building a miniature but architecturally complete version of it from scratch.

The key property is that Mistral never touches the database directly and can never hallucinate data. It can only answer based on what your database actually contains, returned through controlled tool calls that Spring Boot executes.

The 6 tools exposed to Mistral

Spring Boot describes these tools to Mistral at the start of every conversation. Mistral reads the descriptions and decides which one to call. Spring Boot executes only the chosen tool — never arbitrary SQL, never anything outside this defined set. This makes the chatbot safe by design.

The system prompt sent to Mistral

At the start of every conversation, Spring Boot sends Mistral a system prompt that establishes context. This prompt is what enables Mistral to understand your specific data schema and answer questions accurately:

You are a cybersecurity analyst assistant for the SecureLens SOC platform.
You have access to the following real-time data through tool calls:

DATABASE SCHEMA CONTEXT:
- events table: raw log lines from SSH_AUTH, FIREWALL, APACHE, WINDOWS sources
- alerts table: detected security incidents with MITRE ATT&CK tags and severity
- ioc_cache table: threat intelligence results from VirusTotal, AbuseIPDB, Shodan, NVD, OTX

BEHAVIOR:
- Always use tool calls to fetch real data before answering questions about logs or alerts
- Never invent or estimate data — only state what the database returns
- Be concise and actionable — analysts need fast answers
- When listing IPs or events, always include counts and timestamps
- If a question requires multiple data sources, make multiple tool calls
- Current time context: {current_timestamp}

Respond in clear plain language. Use bullet points for lists. Bold important values.

The system prompt per conversation turn

After the system prompt, each conversation turn follows this structure. Spring Boot manages this loop automatically in ChatService:

1. Append analyst message to conversation history
2. Send history + tool definitions to Mistral (call 1)
3. If Mistral returns a tool_call:
     a. Parse tool name and parameters
     b. Execute the corresponding repository query
     c. Append tool result to conversation history
     d. Send updated history to Mistral (call 2)
4. Return Mistral's final text response to the React chat panel

Integration into the existing architecture

The chatbot adds exactly two new components to the backend and one to the frontend. Nothing existing is modified:

New componentOwnerWhat it does
ChatController.javaDev 2POST /api/chat — receives message + history, returns answer
ChatService.javaDev 2Manages two-call Mistral pattern, tool execution, history
ChatPanel.tsxDev 2Slide-in chat UI, message history, markdown rendering

The ChatService calls existing repositories directly — EventRepository, AlertRepository, and IocCacheRepository. It reuses the existing MistralService HTTP client already built for alert triage. No new tables, no new dependencies beyond a markdown renderer on the frontend.

Why this is architecturally significant: The chatbot transforms SecureLens from a dashboard that presents data to an interface that understands intent. The same data is now accessible in two modes — structured (pages and tables) and conversational (natural language). This is the architectural pattern behind Microsoft Copilot for Security, Elastic AI Assistant, and Splunk AI. You build it from first principles in two components.
Fallback behavior: If Mistral's API takes longer than 10 seconds, ChatService executes the most likely tool directly based on keywords in the analyst's message and returns the raw query result as a formatted table. The chatbot is never blocked by API latency.

New API endpoint

MethodEndpointDescription
POST /api/chat Body: { message: string, history: Message[] } — Returns: { answer: string, toolCalled: string, dataSnapshot: object }

The history array contains the previous messages of the current session. The frontend maintains this array in React state and sends it with every new message. This gives Mistral full conversation context — the analyst can ask follow-up questions that reference earlier messages.

The dataSnapshot in the response contains the raw data that was fetched from the database, so the frontend can optionally render it as a table alongside the natural language answer.
